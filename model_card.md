## 1. Base Project

The foundation of this project is **Game Glitch Investigator** (Modules 1–3): a Streamlit number-guessing game intentionally seeded with six bugs. Over the first three modules I used Claude (Anthropic) to locate and fix every bug, then refactored the game logic into `logic_utils.py` and wrote 11 pytest cases to verify each fix. The original files (`logic_utils.py`, `tests/test_game_logic.py`) are fully preserved and still pass.

This final version (Module 4+) builds on top of that foundation by adding AI-powered features: an LLM-narrated auto-solver, game analysis coaching, AI hints, a RAG-based Strategy Advisor, and a reliability test suite.

---

## 2. AI Collaboration

### How AI was used to build this project

**Claude (claude-sonnet-4-6 via Claude Code)** was the primary AI collaborator throughout:

- **Bug fixing (Modules 1–3):** Claude identified all six bugs (swapped hint directions, broken scoring, missing range validation, broken game reset, etc.) and proposed targeted one-line fixes. I verified each fix against the test suite before accepting.
- **Code structure:** Claude suggested extracting pure game logic into `logic_utils.py` so the UI layer could stay thin. I accepted this because it made the code easier to test independently.
- **AI feature design:** I described what I wanted (an agent that "shows its work" while solving the puzzle), and Claude proposed the Plan → Act → Check → Reflect loop pattern. I approved the design and reviewed every method before it was written.
- **RAG Strategy Advisor:** Claude wrote the `StrategyRAG` class following the same TF-IDF + Ollama pattern I already understood from reviewing `ai_engine.py`. I reviewed the chunking logic, the prompt template, and the fallback behavior before accepting.
- **Tests:** All 33 tests were written by Claude with my direction on what behaviors to cover. I ran them locally and flagged any test that seemed to over-specify implementation details.

### What I reviewed and changed

- I rejected an early version of the auto-solver that had the LLM choose the next guess — I required the solver to always use binary search deterministically, with the LLM only narrating.
- I specified that all AI features must have a working fallback so the app never crashes when Ollama is offline.
- I directed that the Strategy Advisor use a built-in knowledge base rather than user-uploaded documents, keeping the feature relevant to the game.

## 5. Potential Biases and Limitations

### LLM output variability
The hints, analysis, and strategy advice are generated by `llama3.2`, a general-purpose LLM. Responses are non-deterministic: running the same prompt twice may produce different wording. The auto-solver's *logic* (binary search) is deterministic, but its *narration* varies.

### Strategy advice grounding
The RAG system retrieves from a hand-written knowledge base (`STRATEGY_KB` in `strategy_rag.py`). The content reflects standard binary search theory and is accurate for this game. However, it was written by me and reviewed by Claude — it has not been independently validated by a domain expert. Any errors in the KB propagate directly into the advice.

### Hint direction validation is keyword-based
`game_guardrails.py` checks that hints contain directional words ("lower", "higher", etc.). A hint that uses unusual phrasing (e.g., "aim for a smaller value") could pass the keyword check even if it is confusing to players.

### Evaluation keywords are brittle
The `strategy_evaluator.py` test cases check for specific keywords (e.g., "binary", "51", "careful"). If Ollama paraphrases correctly but uses different words, a test case will fail even though the advice is sound. This is a known limitation of keyword-based eval.

## 7. Reflections on AI Collaboration

**What worked well:**  
Using Claude to write boilerplate (dataclasses, test fixtures, Streamlit tab structure) saved significant time. Claude was also good at catching edge cases I hadn't thought of — like the zero-norm TF-IDF vector guard in the retrieval step.

**What required careful human judgment:**  
Every design decision about *what* the AI should do — not *how* — required me to think carefully. Should the auto-solver use the LLM to pick guesses, or just to narrate? Should the strategy advisor accept user-uploaded documents, or use a built-in corpus? Claude presented options, but I had to choose based on the course requirements and my understanding of reliability tradeoffs.

**Surprising finding:**  
The LLM's narration in the auto-solver made the biggest impact on the feel of the app. Without it, watching a binary search is dry. With it, each step reads like a character making a decision. This was the most unexpected value-add from the generative AI layer.

**What I would do differently:**  
I would use semantic embeddings (e.g., sentence-transformers) instead of TF-IDF for the strategy RAG, which would handle paraphrased queries better. TF-IDF struggles when the player's question uses different vocabulary than the knowledge base (e.g., "divide in half" vs. "binary search").

**On AI-assisted coding:**  
Working with Claude Code made me more intentional about code review. Because I wasn't typing every line myself, I had to be more deliberate about *reading* every line — understanding what it did before accepting it. That habit of careful review was the most valuable skill I developed through this process.
